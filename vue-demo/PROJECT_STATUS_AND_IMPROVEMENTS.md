# 项目功能清单与改进建议

本文档基于当前仓库实际代码整理，目的不是替换原有部署文档，而是作为一份“现状说明 + 功能总览 + 待改进项”记录。

## 1. 项目概述

这是一个基于 `Vue 3 + Vite + Element Plus + Pinia + Vue Router` 的家庭任务管理系统，后端为 `Spring Boot` 风格接口，围绕“家庭成员协作、任务分配、状态流转、提醒、统计、个人中心”等场景展开。

系统采用前后端分离架构：

- 前端负责交互、布局、状态展示与表单提交
- 后端负责认证、家庭数据、任务数据、权限校验和持久化

## 4. 当前后端接口能力

### 4.1 任务相关

- `POST /api/tasks` 新建任务
- `GET /api/tasks` 获取任务列表
- `GET /api/tasks/{id}` 获取任务详情
- `PATCH /api/tasks/{id}/status` 修改任务状态
- `DELETE /api/tasks/{id}` 删除任务
- `POST /api/tasks/{id}/comments` 新增评论
- `GET /api/tasks/{id}/comments` 获取评论列表
- `GET /api/tasks/{id}/activities` 获取操作日志

### 4.2 家庭相关

- `GET /api/families/me` 获取当前家庭
- `GET /api/families/me/members` 获取当前家庭成员
- `PATCH /api/families/me/name` 修改家庭名称
- `PATCH /api/families/me/members/{userId}/role` 修改成员角色
- `PATCH /api/families/me/members/{userId}/identity` 修改成员身份
- `DELETE /api/families/me/members/{userId}` 删除成员
- `POST /api/families/me/exit` 退出家庭

### 4.3 认证相关

- `POST /api/auth/login`
- `POST /api/auth/register`
- `POST /api/auth/reset-password`

## 5. 当前与文档或预期相比的主要差异

### 5.1 前端接口地址仍然写死

当前前端请求封装仍使用：

- `http://localhost:8081/api`

这意味着：

- 本地开发可用
- 但上线时更适合改为环境变量或相对路径 `/api`
- 目前项目还未做完整的环境分层配置

### 5.2 任务负责人选择依赖家庭成员接口

当前任务创建弹窗负责人来自：

- `/families/me/members`

这比早期写死“爸爸 / 妈妈 / 孩子”要正确，但也带来一个要求：

- 如果后端成员接口异常，负责人下拉会为空
- 前端需要和后端成员数据结构保持一致

### 5.3 任务权限已经调整

当前后端任务权限已经放开：

- 家庭成员可修改任务状态
- 家庭成员可删除任务
- 创建任务时负责人必须属于当前家庭

因此，任何“仅 OWNER 才能操作任务”的旧说明都已经过时。

### 5.4 当前没有全局后台管理页

之前临时加过一个“后台管理”页面，但已撤回。

现状是：

- 只有家庭空间级别的管理
- 没有真正的全局管理员后台
- 没有全站用户 / 全站家庭的统一管理界面

## 6. 当前后端实现特点

### 6.1 权限控制

后端当前采用了家庭维度的权限判断：

- 用户必须属于某个家庭，才能访问任务和家庭相关数据
- 创建任务时，负责人必须是当前家庭成员
- 任务状态和删除操作已经面向家庭成员开放

### 6.2 任务与家庭绑定

- 任务列表按家庭加载
- 任务创建时自动绑定家庭 ID
- 家庭成员信息会影响任务负责人展示

### 6.3 统一返回结构

后端当前通过统一响应体返回数据，前端通过 `request` 封装进行解析。

不过前端有些地方仍保留了兼容式取值方式，说明接口结构在历史上可能发生过变化。

## 7. 目前需要改进的内容

### 7.1 部署与环境配置

- 将前端请求基地址改为环境变量控制
- 开发 / 生产环境分离配置
- 补充后端端口、代理、跨域说明
- 统一本地开发与部署说明

### 7.2 权限体系

- 当前任务权限已经放开到家庭成员级别
- 但家庭管理类操作仍建议保留更严格的权限控制
- 建议补一份明确的权限矩阵，避免前后端理解不一致

### 7.3 全局后台管理

如果项目目标里包含“真正的后台管理”，建议新增：

- 全局用户列表
- 全局家庭列表
- 按家庭 ID / 用户 ID 查询
- 查看成员归属家庭
- 管理员操作日志
- 权限分级与管理入口

### 7.4 接口返回结构统一

目前前端部分接口处理里，存在对 `res.data` / `res.data.data` 的兼容式写法。
建议后端统一返回结构，例如：

- `code`
- `msg`
- `data`

并确保前端所有页面都按照同一层级取值。

### 7.5 任务列表与成员映射

建议继续统一以下映射逻辑：

- `userId -> nickname`
- `familyIdentity -> 中文身份名`
- `roleCode -> 中文角色名`

避免任务中心、任务详情、家庭空间里展示不一致。

### 7.6 数据校验与空状态

建议增强：

- 无家庭成员时的提示
- 无任务时的提示
- 无权限时的提示
- 接口失败时的降级展示
- 加载中状态与错误重试入口

### 7.7 代码可维护性

目前部分页面逻辑较集中，后续可以考虑：

- 将成员信息抽成共享 composable
- 将任务状态和成员展示逻辑抽成工具函数
- 将表单弹窗拆成更小组件
- 抽离重复的接口取值逻辑

- 任务页结构更完整

因此，任何旧的部署说明、功能说明或权限说明，都需要以当前代码为准重新整理。
